### Profile 分工

| Profile | 职责 | 不负责 |
|---------|------|--------|
| **default（毛球）** | 世界书制作、world-tree skill 修改/优化、控制台维护 | 启动游戏、运行叙事 |

## 二、控制台（world-tree-console.html）

### 加载

浏览器直接打开 `world-tree-console.html`。

**方式一：选择目录** — 选择 `D:/world-tree-data` 目录，控制台读取全部模组数据。

**方式二：选择世界书文件（v12.16 新增）** — 点击「📄 选择世界书文件」，直接选 `.json` 文件。概览 Tab 显示世界书条目摘要：条目数、类型分布、前15条触发词+内容预览。切到「世界书」Tab 查看完整条目列表。适合快速浏览已生成的世界书。

**方式三：选择角色卡文件（v12.19 新增）** — 选 `.json` 或 `.png`（ST格式）角色卡文件。控制台自动识别格式并打标签【角色卡】，显示角色名+来源格式+关键标签。加载后以角色卡驱动模式（M19）运行。详见 `references/M19-角色卡驱动.md`。

**自动识别机制（v12.19）：** 所有 `.json` 文件加载时自动扫描结构——检测到世界书特征（spec/entries）→【世界书】标签；检测到角色卡特征（name+first_mes+personality）→【角色卡】标签。两种标签不同颜色/图标区分。

### 对话面板配置

1. 切到「对话」Tab
2. 连接设置：
   - API根地址: `http://127.0.0.1:18742`
   - API Token: 从 `platforms.api_server.key` 获取
3. 点「测试连接」→ 显示"已连接"
4. 开始对话

### 对话中的自动行为

- `[世界树:模组名]` 标签高亮渲染
- 每次DM回复后自动刷新控制台数据
- 聊天历史按模组隔离（localStorage key: `wt-chat-{moduleName}`）
- Hermes 远端 session 按模组隔离（`sessionsByModule`），避免不同世界书共用同一个对话上下文
- 「开启新对话」会立即为当前模组创建新的 Hermes session；只清空聊天上下文，不清空世界树进度文件
- 会话管理器会为当前模组保存最近使用过的 session，可手动切换、复制 session id、忘记绑定
- 新 session 首次发送消息时会自动附带“启动包”，包含当前模组、分支、场景、摘要链和待决问题
- 「体检」Tab 可检查 API 连接、目录记忆能力、当前控制台 HTML hash 与主 / writer 镜像目标路径
- 写作剪贴板：DM消息旁「保存片段」→ 右侧面板查看/复制

### API 端点

当前稳定版使用非流式对话接口：

```
控制台 HTML → fetch /chat → http://127.0.0.1:18742 → writer gateway → agent loop
```

- `/health`：健康检查
- `/api/sessions`：创建会话
- `/api/sessions/{id}/chat`：非流式对话
- `/api/sessions/{id}/chat/stream`：SSE 流式接口保留为待恢复能力，当前控制台不依赖它

## 三、常见问题

### API Server 连接失败

检查：
1. 端口是否正确（`platforms.api_server.extra.port`，不是 `api_server.port`）
2. 端口是否被占用：`netstat -ano | grep 18742`
3. gateway 是否在运行：`hermes -p writer gateway status`
4. 重启 gateway：`hermes -p writer gateway restart`

### 默认 profile 的 API Server 占用了 8642

两个 gateway 使用不同端口即可：
- default → 8642
- writer → 18742

互不冲突。

### 切换世界书后出现旧世界书的内容（串戏）

**这是控制台最常见的上下文污染问题。**

历史原因：同一个 Hermes 会话中先后加载两个不同世界书时，Hermes 会话历史不会自动清空，旧世界书的对话记录会保留在上下文中，导致加载新世界书时模型把旧设定带进来。

**当前解决方法：** 控制台已按模组隔离本地聊天记录和远端 Hermes session。直接输入「加载万界」或 `/引擎 load 万界` 时，控制台会先切换到目标模组并清掉该模组旧 session 绑定。需要主动清脑时，点「开启新对话」；它会立即创建新的远端 session，但不会删除 `runtime.json`、`canon_state.json`、`scene_chain.json`、存档或分支进度。当前模组键会结合 `storagePath` / `storageKey` 生成稳定指纹，避免两个同名世界书落进同一个聊天桶。

**验证：** 检查 writer profile 的 `agent.log`，搜索同一 `session_id` 下是否出现多个不同模组加载指令。如果是 → 确认是会话未隔离导致的。

详细排查步骤见 world-tree skill 的 `references/控制台会话隔离排查.md`。

两个位置需同步（Windows 路径）：
```
C:\Users\Lenovo\AppData\Local\hermes\skills\creative\world-tree\world-tree-console.html  ← 源文件
C:\Users\Lenovo\AppData\Local\hermes\profiles\writer\skills\creative\world-tree\...       ← profile副本

cp 源文件 → profile副本
```

同步不只限于控制台。`SKILL.md` 与 `references/` 也必须同步到 writer profile，否则 writer 会按旧规则执行。
