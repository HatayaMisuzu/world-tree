# 跨文件一致性审计方法

> 用于世界树多文件技能体系（SKILL.md + 40+ references + SOUL.md + writer档案）的版本一致性检查。

## 审计步骤

### 第1层：SKILL.md 主干自检
1. 检查目录编号：`grep -n "^[0-9]\+\." SKILL.md` — 确认无重复无缺号
2. 检查路径：`grep "~/.hermes\|/home/" SKILL.md` — Windows 下应为 `C:\Users\...\AppData\Local\hermes\`
3. 检查模块映射表：对比模块列表与 `world_type_modules` 中的 `M15a/M15b/M15c` 命名一致性
4. 检查版本号：yaml frontmatter `version:` 与 `# 世界树引擎 vX.X` 标题一致

### 第2层：SOUL.md vs SKILL.md 交叉
1. 指令清单验证：SOUL.md 中列出的每个 `/指令` 必须在 SKILL.md §5 或 references 中存在
2. 流程一致性：DM 工作循环的步骤数、叙事五层顺序是否与 SKILL.md §3.2 一致
3. 色色隔离边界：SOUL.md 的暧昧阈值定义需与 world-tree §6 和 erotic-writing trigger 对齐
4. 信息层级：confirmed/implied/proposed/conflict/open question 五级定义一致

### 第3层：config.yaml vs SOUL.md 交叉
1. `personality:` 默认值应为 `writer`（不是 kawaii）
2. `agent.personalities.writer` 描述应引用 SOUL.md 版本号
3. `agent.personalities.writer` 路径引用应为 Windows 格式
4. `display.personality:` 应与 `personality:` 一致（都指向 writer）

### 第4层：references 版本漂移检测
1. 读每个 reference 文件的头部版本标注（`> 版本: vX.X`）
2. 与 SKILL.md 当前版本比对
3. 重点关注：规范真相源.yaml、M-引擎隔离架构.md、存档分支.md（这三个最容易滞后）

### 第5层：writer 副本同步验证
1. `diff shared_SKILL.md writer_SKILL.md` — 应完全一致
2. `diff shared_真相源.yaml writer_真相源.yaml` — 应完全一致
3. 逐个 reference 文件对比（重点关注本次修改过的）
4. 同步命令：`cp shared/skills/... writer/skills/...`

### 第6层：基础设施验证
1. writer gateway：`curl http://localhost:18742/health`
2. 数据目录：`ls D:/world-tree-data/`
3. 控制台文件：`ls .../skills/.../world-tree-console.html`
4. SOUL.md：确认存在且非空

## 常见缺陷模式

| 缺陷 | 检测命令 | 影响 |
|------|---------|------|
| 步骤编号重复/缺号 | `grep "^[0-9]\+\."` | AI执行流程时跳步 |
| 幽灵指令 | `grep "/xxx" SOUL.md` → 在 skill 中搜索 | AI调用不存在的命令 |
| Unix 路径残留 | `grep "~/\." SKILL.md` | Windows 上 AI 迷路 |
| 版本漂移 | 读 reference 头部 `版本:` vs SKILL.md version | 真相源规则被过时版本覆盖 |
| M15a/M15b 命名未更新 | `grep "M15a\|M15b" references/` | v12.7+ 已重命名为 M15/M15c |
| 默认人格错误 | `grep "^personality:" config.yaml` | 新会话不加载 writer DM |

## 修复优先级

- P0：阻塞性（编号崩溃、幽灵指令、路径错误、人格错误）
- P1：一致性（版本漂移、命名漂移、重复内容）
- P2：技术债（次要文档版本标注、非核心 reference 残留旧命名）
