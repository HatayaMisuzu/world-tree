# 处理/补全引擎 — 世界树前端完整预处理系统

> Processing & Completion Engine v1.0
> 定位：世界树引擎的独立前端。接收用户原始设定，经过数据采集→点子发现→补全提案→推理验证→多维度处理流水线→用户判断，输出完整可用的世界设定包。

## 一、素材库（数据环）

### 职责

存储用户投喂的原始素材和联网搜集的外部素材，为发现引擎提供数据源。

### 数据来源

| 来源 | 说明 | 权重标记 |
|------|------|---------|
| 用户直接投喂 | 原始设定/大纲/人设/片段 | 最高优先级 |
| 联网搜集 | GitHub/Wiki/社区相关设定 | 按权重排序 |
| 历史积累 | 之前处理过的相似世界设定 | 参考标记 |

### 指令

```
/引擎 接收 [文本/文件/链接]    ← 投喂素材
/引擎 搜集 [关键词/维度]        ← 联网搜索
/引擎 素材库                    ← 查看当前素材库
```

---

## 二、发现引擎（点子匹配）

### 职责

从素材库中发现与用户原始设定相似、互补、可借鉴的点子。

### 匹配维度

| 维度 | 说明 | 示例 |
|------|------|------|
| 同类型设定 | 相似世界观中的设定方案 | 用户写了魔法体系→找其他魔法体系的要素 |
| 冲突解决方案 | 同类矛盾在已有设定中怎么解决的 | 时间悖论怎么处理 |
| 空白补全 | 用户没写但同类设定都有的要素 | 写了种族没写种族关系 |
| 热点趋势 | 当前热门/讨论度高的设定方向 | 当前流行什么样的世界观 |
| 独特创新点 | 素材库中特别亮眼的设定 | 某个来源有独特的魔法代价机制 |

### 发现报告格式

```
═══ 点子发现 ───
来自: [来源名称] (复现率: X次 | 热度: ★★★★☆)
相似度: [与用户设定的匹配程度]

原文要点:
> "XXX"

可借鉴点:
- [具体可以怎么用到用户的设定里]
- [为什么这个点子好]

建议位置: [建议插入用户设定的哪个部分]
═══
```

### 指令

```
/引擎 发现                ← 运行点子发现
/引擎 发现 [维度]          ← 在特定维度发现点子
```

---

## 三、补全提案

### 职责

基于发现引擎的结果，对用户的原始文本提出具体的补充和修改建议。

### 提案类型

| 类型 | 说明 |
|------|------|
| 补充提案 | 原始设定缺失但应该有的内容 |
| 优化提案 | 原始设定有但可以更好的内容 |
| 冲突修正提案 | 原始设定中存在矛盾的内容 |
| 创新注入提案 | 从素材库发现的亮点可以融入的内容 |

### 提案格式

每个提案包含：
- 原文位置（引用具体段落）
- 问题描述（为什么需要修改）
- 建议方案（具体怎么改，提供1-3个选项）
- 素材依据（从素材库的哪个来源找到的）
- 影响范围（修改会影响设定的哪些部分）

### 指令

```
/引擎 补全                ← 运行补全提案
/引擎 补全 [提案ID]        ← 查看某个提案详情
/引擎 采纳 [提案ID]        ← 采纳提案
/引擎 拒绝 [提案ID]        ← 拒绝提案
```

## 四、推理验证

### 职责

将补全后的设定在脑中推演一遍，验证其合理性。

### 推演维度

| 维度 | 推演内容 |
|------|---------|
| 时间推演 | 设定中的事件按时间顺序推演，检查前后一致性 |
| 因果推演 | 每个事件的原因→结果链是否完整 |
| 角色推演 | 角色在设定规则下的行为是否合理 |
| 世界推演 | 设定规则长期运行后的世界状态 |
| 交互推演 | 不同设定要素之间的相互作用 |

### 推演报告

```
═══ 推理验证 ───
推演路径: [时间/因果/角色/世界/交互]
推演结果: ✅通过 / ⚠️发现潜在问题 / ❌发现逻辑断裂

如果⚠️/❌:
  问题点: [具体描述]
  断点位置: [在设定的哪个环节出现问题]
  建议修复: [如何修复]
```

### 指令

```
/引擎 推理                ← 运行推理验证
/引擎 推理 [维度]          ← 在特定维度推理
```

## 五、七步处理流水线

### 流程

补全+推理通过后，进入七步处理流水线。每步产生评分和问题清单。

```
补全后的设定 → ①逻辑 → ②矛盾 → ③流畅 → ④创新 → ⑤观感 → ⑥整体 → ⑦综合
```

### 5.1 逻辑检查

检查设定的内在逻辑性。

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 因果完整性 | 每个事件是否有可靠的原因 | 0-10 |
| 规则自洽 | 设定的规则互相不矛盾 | 0-10 |
| 推理闭合 | 设定的结论能从前提推出 | 0-10 |
| 边界清晰 | 规则的边界条件是否明确 | 0-10 |

输出：逻辑评分(满分40) + 问题清单

### 5.2 矛盾检测

检查设定中的明显矛盾。

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 设定内矛盾 | 同一概念不同地方描述不同 | 0-10 |
| 跨维度矛盾 | 维度A和维度B的设定互斥 | 0-10 |
| 数值矛盾 | 时间/距离/数量前后不一致 | 0-10 |
| 隐含矛盾 | 不直接冲突但逻辑推断后矛盾 | 0-10 |

输出：矛盾评分(满分40) + 矛盾清单+解决建议

### 5.3 流畅度检查

检查设定的可读性和流畅性。

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 信息密度 | 每个段落的信息量是否合适 | 0-10 |
| 递进自然 | 设定展开的顺序是否自然 | 0-10 |
| 冗余度 | 是否有重复或冗余内容 | 0-10 |
| 衔接平滑 | 不同设定块之间的过渡 | 0-10 |

输出：流畅度评分(满分40) + 改进建议

### 5.4 创新性评估

评估设定是否有新意——不是完全原创才叫好，合理的创新就有价值。

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 独创度 | 核心设定有多少是独特的 | 0-10 |
| 组合创新 | 已有元素的组合方式是否有新意 | 0-10 |
| 意外感 | 是否有让人意想不到的设定 | 0-10 |
| 反套路 | 是否有意识地打破常规 | 0-10 |

输出：创新评分(满分40) + 亮点标注+可加强点

### 5.5 读者观感模拟

从读者视角模拟阅读体验。

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 吸引力 | 设定是否能让人想看下去 | 0-10 |
| 理解负担 | 理解设定需要多少认知成本 | 0-10 (越低越好) |
| 情感共鸣 | 设定是否能引发情感反应 | 0-10 |
| 记忆点 | 有多少让人印象深刻的地方 | 0-10 |

输出：观感评分(满分40) + 观感建议

### 5.6 整体性检查

检查设定各部分是否构成有机整体。

| 检查项 | 说明 | 评分 |
|--------|------|------|
| 主题统一 | 各部分是否围绕同一核心主题 | 0-10 |
| 风格一致 | 设定风格是否通篇一致 | 0-10 |
| 比例协调 | 各维度的展开比例是否合理 | 0-10 |
| 世界呼吸感 | 设定是否有"活的世界"的感受 | 0-10 |

输出：整体评分(满分40) + 整体建议

### 5.7 综合评分

七步的汇总和建议。

```
═══════ 处理流水线报告 ═══════

① 逻辑检查:   XX/40  [████████░░]
② 矛盾检测:   XX/40  [████████░░]
③ 流畅度:     XX/40  [████████░░]
④ 创新性:     XX/40  [████████░░]
⑤ 读者观感:   XX/40  [████████░░]
⑥ 整体性:     XX/40  [████████░░]
⑦ 综合评分:   XX/280 [████████░░]

强度项: [得分最高的2项]
薄弱项: [得分最低的2项]
优先处理: [薄弱的处理建议]
```

### 指令

```
/引擎 处理                  ← 运行全部七步流水线
/引擎 处理 [步骤名]          ← 运行特定步骤
/引擎 处理 报告              ← 查看上次处理报告
```

## 六、用户判断与修改

### 职责

所有引擎输出都是建议。用户做最终判断和修改。

### 交互模式

```
毛球输出处理报告
  ↓
用户：
  /引擎 采纳 [提案ID]     ← 采纳补全提案
  /引擎 拒绝 [提案ID]     ← 拒绝
  /引擎 修改 [位置] [内容] ← 手动修改设定
  /引擎 重处理            ← 修改后重新运行处理
  ↓
毛球更新设定并重新推理
  ↓
用户满意 → /引擎 打包交付
```

### 修改追踪

每次用户修改记录到 change_log.json，支持撤销。

## 八、数据目录

所有数据统一存放于独立目录，不污染世界树引擎数据。

```
D:/world-tree-data/_processing_engine/
├── material_pool/           ← 素材库
│   ├── user_input/          ← 用户投喂
│   ├── web_collected/       ← 联网搜集
│   └── pool_index.json
├── discoveries/             ← 发现引擎输出
├── proposals/               ← 补全提案
├── reasoning/               ← 推理验证结果
├── pipeline_reports/        ← 七步处理报告
├── change_log.json          ← 用户修改记录
├── current_draft/           ← 当前版本设定
└── packages/                ← 打包好的设定包
```
