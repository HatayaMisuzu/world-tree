# 外部引擎移植架构

> 将 world-tree skill (v12.19) 完整移植到独立桌面应用（无 Hermes Agent、直连 LLM）时的架构参考。
> 本文档记录核心架构设计决策，供外部引擎实现时参考。

## 适用场景

- 将世界树引擎**脱离 Hermes Agent** 运行
- 桌面应用/移动端/Web 端直连任意 OpenAI-compatible LLM
- 需要完整保留 M1-M19 模块系统的行为
- 需要独立持久化和管理 world-tree-data

## 总体架构

外部引擎分为两层：**桌面端（代码层）** 和 **LLM 端（prompt 层）**。

```
┌───────────────────────────────────────────┐
│  桌面端 (JS/Electron/Node.js)               │
│                                            │
│  • 文件 I/O（读写 world-tree-data）         │
│  • 引擎状态机（启动/加载/运行/暂停/退出）    │
│  • 执行确定性规则（守门人/budget/世界书匹配） │
│  • 构建 engine packet → 发送给 LLM          │
│  • 解析 LLM 结构化输出 → 回写状态           │
└──────────────────┬────────────────────────┘
                   │ engine packet + 用户输入
                   ▼
┌───────────────────────────────────────────┐
│  LLM 端 (GPT/Claude/DeepSeek 等)           │
│                                            │
│  • M13 五层叙事生成（角色→环境→剧情→语气→记忆）│
│  • 角色扮演与对话                           │
│  • M15 规则可行性自检                       │
│  • M15c 叙事质量自审                       │
│  • M18 场景走向预测                        │
│  • 输出结构化段（叙事+状态变更+世界书提案）   │
└───────────────────────────────────────────┘
```

## 关键架构决策

### 一、知识注入：四层架构

以 Hermes 内运行时，毛球通过 `知识库索引.md` 的关键词映射按需加载 50+ 参考文档。在外部引擎中，这个策略需要重构。

#### 第一层：JS 硬编码规则（代码层）

> 确定性逻辑用代码执行，不进 LLM prompt。

| 模块 | 实现位置 | 职责 |
|:---|:---|:---|
| 守门人 | `engine/guardian.js` | 角色归属校验、模组隔离、续写安全检查 |
| 上下文预算 | `engine/context-budget.js` | 四档预算计算（宽裕/正常/收紧/紧急） |
| M2 匹配 | `data/worldbook.js` | 关键词触发、语义匹配、条目排序 |
| M11 场景链 | `data/scenes.js` | 摘要追加、旋转、压缩 |
| M15 规则审核 | `data/rules.js` | 行为可行性检查（物理/魔法/社会规则） |
| 输出解析 | `engine/output-parser.js` | 解析 LLM 回复中的结构化段，回写状态 |

Desktop 在构建 engine packet 时，直接注入这些检查的**结果**（而非规则原文）：
```json
{
  "guardianPass": true,
  "budget": "normal",
  "matchedWorldbook": 3,
  "sceneChain": 5
}
```

#### 第二层：模块知识卡（prompt 内）

每个模块浓缩为 **300-800 tokens** 的知识卡 JSON，只注入当前激活模块的。

知识卡格式：
```json
{
  "moduleId": "M13",
  "name": "叙事引擎五层",
  "rules": [
    "每轮输出按五层构建：角色层→环境层→剧情层→语气层→记忆层",
    "角色层：当前角色状态、情绪、动机",
    "环境层：场景描写、感官锚点（至少2种感官）"
  ],
  "outputImpact": "影响叙事文本的段落结构和内容密度"
}
```

**不放进知识卡的内容：** 参数细节、指令语法、技术实现、边缘案例完整列举。

知识卡从原始 reference 文档用 LLM 浓缩生成，人工审核后固化。每张卡记录 `sourceHash` 用于后续同步检测。

按 epic 预设（16 模块激活），实际有叙事影响的模块 6-8 个，知识卡层约 **4,000 tokens**。

#### 第三层：全文知识库（按需检索）

50+ 原始 markdown 文档全文保存在 `defaults/engine-profile/knowledge/` 下。LLM 遇到知识卡未覆盖的问题时，通过关键词检索获取全文。

```javascript
// engine packet 末尾提示
`【规则检索说明】
如果你遇到上述知识卡未覆盖的规则问题，
请回复：/检索 [关键词]
桌面端会自动检索完整文档并注入下一轮 prompt。`
```

现有 `engineKnowledgeSearch`（Electron IPC）提供关键词匹配+上下文摘要。

#### 第四层：内置帮助文档（不进 prompt）

命令详解、快速上手、创作方法论等参考文档做成 Desktop 内置帮助面板，用户自己查阅。

#### 各层 token 对比

| 层 | 内容 | Token |
|:---|:---|:---:|
| Layer 1 | JS 执行结果（简短结构化数据） | ~200 |
| Layer 2 | 激活模块知识卡（6-8 模块） | ~4,000 |
| Layer 3 | 按需检索（0 ~ 3,000） | 0~3,000 |
| Layer 4 | 不进 prompt | 0 |

### 二、数据写入：三级受控写入

"完整复制 skill" 要求外部引擎能写入 world-tree-data（/存档、/世界书 add、/场景 move 等都需要写），但写入需要分级保护。

#### 写入模式检测

```javascript
if (hermesAPI 可达) {
  // 委托写入：通过 Hermes API 写，防竞争条件
  delegateWriteToHermes(data);
} else {
  // 本地受控写入：Desktop 自己写
  controlledLocalWrite(data, level);
}
```

#### 三级写入权限

| 级别 | 操作示例 | 安全等级 |
|:---|:---|:---:|
| **AUTO** | 场景摘要追加、runtime 位置更新、存档创建、proposed canon 更新、角色状态变更 | 🟢 自动执行 |
| **CONFIRM** | 创建角色、修改世界书条目、确认 canon（proposed→confirmed）、删除存档/分支 | 🟡 弹框确认 |
| **MANUAL_ONLY** | 覆盖 canon confirmed 字段、修改 _engine/ 配置 | 🔴 不自动写，输出 proposed patch |

用户在 Power User 面板可调整每类操作的默认级别。

#### 写入保护机制

每次写入执行六步保护：

```javascript
async function writeJson(filePath, data, level) {
  // 1. 路径守卫 — 禁止越界到 _engine/ 或其他模组
  guardPath(filePath);
  
  // 2. 级别检查 — 匹配当前写入策略
  checkLevel(level);
  
  // 3. 自动备份 — 写入前备份当前文件
  await backupFile(filePath);
  
  // 4. JSON 校验 — 写入前验证
  JSON.parse(JSON.stringify(data));
  
  // 5. 原子写入 — 先写 .tmp 再 rename
  await fs.writeFile(filePath + ".tmp", raw, "utf8");
  await fs.rename(filePath + ".tmp", filePath);
  
  // 6. 读回验证
  validateStructure(filePath, await fs.readFile(filePath, "utf8"));
}
```

#### 回滚机制

每次写入前自动备份到 `.bak.{timestamp}`，支持：

```
/回滚 runtime.json           → 恢复上一个备份
/回滚 runtime.json --list    → 列出所有备份
/回滚 runtime.json --index 2 → 恢复到指定版本
```

#### 写入签名

每次写入在目标 JSON 中留下签名，供 Hermes 加载时检测冲突：

```json
{
  "_writtenBy": "world-tree-desktop",
  "_writtenAt": "2026-06-04T14:30:22.000Z",
  "_engineVersion": "world-tree-v12.19-embedded"
}
```

### 三、17 步顺序生命周期（替代子 agent 调度）

Hermes 环境下使用 delegate_task 并行处理。外部引擎没有子 agent 能力，改为顺序阶段式处理：

```
Phase 1 — 安全校验
  1. 守门人校验
  2. 上下文档位检查

Phase 2 — 上下文构建
  3. 注入场景摘要链（N 条，由 budget 决定）
  4. 注入引擎知识卡（仅激活模块）
  5. 检索并注入世界书命中条目
  6. 注入世界状态/组织/角色数据

Phase 3 — 发送 LLM
  7. 构建 engine packet → 发送
  8. 接收回复 → output-parser 解析

Phase 4 — 后处理
  9. 应用状态变更（写入 runtime.json）
  10. 更新 canon_state
  11. 更新场景摘要链
  12. 更新角色认知（M9）
  13. 规则审核记录（M15，非阻塞）
  14. 叙事审查记录（M15c，非阻塞）
  15. 生成场景预测（M18）
  16. 随机事件检查（M17）
  17. 记忆归档（压缩）
```

每个 Phase 可独立开启/关闭，由模块激活状态决定。

### 四、结构化输出协议

LLM 回复不再只是纯叙事文本，而是包含结构化段的混合输出：

```
[叙事正文...]

---STATE---
scene: 王都酒馆
time: 午后
location: 王都酒馆·吧台

---CHARACTERS---
艾琳: mood=好奇, location=王都酒馆, status=正常
旅人: location=王都酒馆, status=正常

---CANON---
confirmed:
  - "艾琳是酒馆老板的女儿"
implied:
  - "旅人似乎认识艾琳的父亲"

---MEMO---
- 旅人暗示了与艾琳父亲的旧识关系

---SCENE_PREDICTION---
next:
  - "旅人说出更多关于父亲的事"
  - "艾琳追问旅人的身份"
```

每段均为可选。Desktop 的 `output-parser.js` 用正则可 `/^---(\w+)---$/` 匹配段分隔符，解析后分别写入对应文件。

## Hermes 共存策略

| 使用场景 | 写入策略 |
|:---|:---|
| Desktop 直连 LLM（无 Hermes） | 本地受控写入 |
| Desktop → Hermes API | 委托给 Hermes 写 |
| 混合使用（上午 Hermes，下午 Desktop） | 不同时运行；启动时检测 `_writtenBy` 判断是否需要重新加载 |

## 与原有功能的关系

- **Hermes API 模式**（chat Tab）：保留不变
- **直接 LLM 模式**（game Tab）：升级为完整引擎
- **数据浏览 Tab**（overview/worldbook/characters）：保留不变
- **卡片导入**：升级——导入的卡片可自动写入选中的模组
- **写入策略**：从"只读+导出"升级为"可控写入"

## 知识卡生成流程

```bash
# 从 world-tree skill 的 reference 文档生成知识卡
scripts/generate-knowledge-cards.js

# 检测知识卡是否需要更新（比较 sourceHash）
scripts/check-knowledge-sync.js
```

知识卡存放路径：`defaults/engine-profile/modules/{moduleId}.json`

原始文档存放路径：`defaults/engine-profile/knowledge/{fileName}.md`
