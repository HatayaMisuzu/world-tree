### 1.3 关键缺口（阻碍普通用户使用的障碍）

| # | 缺口 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | **无安装包** | 🔴 | 用户需要装 Node.js、开命令行、手动运行 `node server.js` |
| 2 | **无新手指引** | 🔴 | 打开浏览器是空白配置页，不知道要填什么 |
| 3 | **错误信息技术化** | 🔴 | `LLM 配置不完整`、`fetch failed`—用户看不懂 |
| 4 | **无数据导出/备份 UI** | 🟡 | 所有数据在本地 JSON，用户不知道怎么备份 |

| 6 | **单窗口限制** | 🟡 | 同时只能聊一个模组，无法多开 |
| 7 | **无 PWA/离线支持** | 🟡 | 浏览器关了数据还在但界面需重新加载 |
| 8 | **对话无搜索/过滤** | 🟡 | 历史消息多了只能滚动翻 |
| 9 | **中英混杂** | 🟡 | 引擎 prompt 中英混杂，部分报错用英文 |
| 10 | **无用户手册** | 🟡 | 只有 AI 开发指南，没有面向用户的帮助文档 |

## 2. 用户旅程分析

### 2.1 目标用户画像

| 画像 | 技术能力 | 使用场景 | 关键需求 |
|------|---------|---------|---------|
| **A: 写作爱好者** | 低 | 写小说、角色扮演 | 开箱即用、界面好看、数据不丢 |
| **B: TRPG 玩家** | 中低 | AI 跑团、剧本杀 | 导入模组、规则一致性、骰子 |
| **C: 创作者** | 中 | 构建世界观、角色设计 | 炼金台、批量导入、多模组管理 |
| **D: 开发者** | 高 | 二次开发、集成 | API 完备、可扩展、文档好 |

### 2.2 当前用户旅程痛点

```
首次使用:
  下载项目 → 装 Node.js? → 翻文档找 npm install → 命令行启动
  → 打开浏览器看到空白配置页 → "API地址是什么？模型是什么？Key在哪？"
  → 配完还要点测试 → 选模组（空的）→ 要先用炼金台创建...
  → 😤 门槛太高，普通用户在这一步就流失了

日常使用:
  每次都要开命令行启动 → 浏览器标签不能关 → 对话多了滚动累
  → 突然报错不知道什么意思 → 想备份不知道数据在哪
  → 😩 体验粗糙，不像一个正经产品
```

## 3. 产品化建议清单

按用户可见影响和实现成本分为四象限：

### 3.1 🔴 P0 — 必须做（影响用户能不能用）

#### 建议 1: 一键安装包（Windows/macOS/Linux）

**现状**：需要手动 `npm install && node server.js`  
**目标**：下载 → 双击 → 浏览器自动打开

- **Windows**: 用 [pkg](https://github.com/vercel/pkg) 或 [sea (Node.js Single Executable)](https://nodejs.org/api/single-executable-applications.html) 打包成 `.exe`
- **macOS**: 打包成 `.app` bundle
- **自动启动浏览器**：`node:child_process` 调用 `start http://localhost:3000`
- **系统托盘图标**：后台运行，托盘退出

```
用户视角的改变:
  之前: 下载 → 装 Node → 解压 → 开 cmd → cd → npm install → node server.js
  之后: 下载 → 双击 WorldTree.exe → 浏览器自动打开 ✅
```

**实现预估**: 3-5 天  
**文件影响**: 新增 `scripts/package.js` + 构建配置

#### 建议 2: 首次运行向导（Onboarding）

**现状**：打开是空白配置页，用户不知道要做什么  
**目标**：第一次打开时引导用户完成三步

```
Step 1: 欢迎页 — "欢迎使用世界树！请输入你的 LLM API Key"
        [API 地址] 默认填入 https://api.deepseek.com/v1
        [模型名]   默认 deepseek-v4-flash
        [API Key]  密码框
        [下一步 →]

Step 2: 创建第一个模组
        "选择你想怎么玩："
        [📖 创建一个新世界] → 命名后进入空世界
        [🃏 用角色卡开始] → 从模板选一个
        [⚗️ 从文档创建] → 打开炼金台

Step 3: 开始对话
        "一切都准备好了！试试输入：你好"
        [简单的示例提示]
```

**实现预估**: 3-4 天  
**文件影响**: `world-tree-console.html` 新增 `showOnboarding()` + CSS

#### 建议 3: 用户友好的错误信息

**现状**：技术栈错误直接抛给用户  
**目标**：每个错误都有「人话解释 + 解决建议」

**需要覆盖的场景**：

| 当前错误信息 | 应改为 |
|-------------|--------|
| `LLM 配置不完整，请检查地址、模型和 API Key` | "还没有连上 AI 模型。请先在首页填写你的 API Key（通常在 AI 服务商的网站可以获取）" |
| `fetch failed` | "无法连接到 AI 服务。请检查：1) 网络是否正常 2) API 地址是否正确 3) 是否需要科学上网" |
| `模组「xxx」已存在` | "这个名称已经被用过了，请换一个名字" |
| `端口 3000 已被占用` | "世界树需要的端口被其他程序占用了。请关掉其他程序后重试" |
| `LLM 调用失败` | "AI 没有正常返回结果，可能是服务暂时拥堵。等几秒再试一次" |

**实现预估**: 1-2 天  
**文件影响**: `server.js` 各 handler 的 errorMsg + HTML 的 createToast 逻辑

#### 建议 4: 数据导出/备份 UI

**现状**：只能手动去硬盘翻 JSON 文件  
**目标**：在界面里一键导出/导入

- **导出模组**：`world.json` + `shared/*` + `runtime/chat.jsonl` → 一个 `.wtpack` ZIP 文件
- **导入模组**：拖拽 `.wtpack` → 自动解压到 `data/engine/worlds/`
- **一键备份**：整个 `data/engine/` → 带时间戳的 ZIP
- **对话导出**：chat.jsonl → `.txt` 或 `.md` 格式（可读性好的对话日志）

**实现预估**: 3-4 天  
**文件影响**: 新增 `src/core/export.js` + HTML 新增「备份」标签页按钮

### 3.2 🟡 P1 — 应该做（显著提升体验）

#### 建议 5: 内置示例/模板

**现状**：新用户首次打开零内容  
**目标**：开箱即玩，有示例模组和角色卡

- **预置角色卡**：2-3 个通用角色（「图书馆管理员」「神秘旅人」等），带完整人格画像
- **示例世界**：1 个微型世界书（主角 + 2 场景 + 几件物品），用户可直接选中开聊
- **「试试看」按钮**：一键加载示例，跳过所有配置步骤

**实现预估**: 1-2 天  
**文件影响**: 新增 `defaults/examples/` 目录，修改首页渲染

#### 建议 6: 对话改进（搜索/上下文/导出）

**现状**：聊天功能基础，缺少专业 IM 体验  
**目标**：让对话像正经聊天应用

- **搜索历史**：`Ctrl+F` 在当前对话中搜索关键词，高亮匹配
- **对话上下文**：鼠标悬停消息显示「这是第几轮」「关联的状态段」
- **对话分支**：在某个消息点「创建分支」→ 走不同的回复 → 可对比
- **消息操作**：每条消息可以 📋复制/⭐收藏/🗑删除/📎引用回复
- **对话导出为小说**：一键把整个对话转成 Markdown 小说格式

**实现预估**: 4-5 天  
**文件影响**: HTML chat 视图 + 新增 `src/core/export-chat.js`

#### 建议 7: 状态面板升级

**现状**：侧栏状态面板功能正确但视觉朴素  
**目标**：实时、可交互、有视觉层次

- **情绪变化动画**：四维条形图用平滑过渡动画展示变化趋势
- **迷你图表**：过去 N 轮的情绪变化曲线（`engagement`/`tension` 折线图）
- **快捷操作**：状态面板内直接点「调高紧张度」「放慢节奏」等快捷指令
- **场景缩略图**：当前场景用卡片展示，包含角色和描述

**实现预估**: 3-4 天  
**文件影响**: HTML + CSS 新增图表组件

#### 建议 8: 配置持久化 + 多配置支持

**现状**：一套 LLM 配置，切换需手动改  
**目标**：保存多套配置，一键切换

- **配置预设**：「DeepSeek 默认」「OpenAI」「本地 Ollama」等
- **按模组记忆配置**：每个模组绑定自己的 LLM 配置
- **配置导入/导出**：分享配置给其他人

**实现预估**: 2 天  
**文件影响**: `server.js` + `config.json` 结构扩展

#### 建议 9: PWA 支持

**现状**：纯网页，必须浏览器开着  
**目标**：可安装为桌面应用，离线可用

- **Service Worker**：缓存 HTML/CSS/JS，断网也能打开界面
- **manifest.json**：应用图标、名称、主题色
- **安装提示**：Chrome 自动弹出「安装世界树桌面」

**实现预估**: 1-2 天  
**文件影响**: 新增 `manifest.json` + `sw.js`

### 3.3 🔵 P2 — 可以做（提升品质感）

#### 建议 10: UI 主题系统

**现状**：只有暗色主题  
**目标**：亮色/暗色 + 自定义强调色

- **亮色主题**：完整亮色 CSS 变量集（纯白/浅灰背景）
- **强调色**：5 种预设色（蓝/紫/绿/橙/粉）
- **用户偏好保存**：写入 `config.json` 的 `theme` 字段

**实现预估**: 2 天  
**文件影响**: HTML CSS 变量扩展

#### 建议 11: 响应式布局

**现状**：`min-width:960px`，小屏幕无法使用  
**目标**：平板和手机也能应急使用

- **断点**：`<768px` 手机布局（单列、折叠状态面板）
- **触控优化**：按钮间距增大、输入框自动调出键盘

**实现预估**: 3-4 天  
**文件影响**: HTML CSS 媒体查询

#### 建议 12: 多语言界面

**现状**：中英混杂  
**目标**：完整中文 + 英文界面，可切换

- **i18n 方案**：前端用简单 `{t('key')}` 函数，字符串集中到 `locales/zh.json` 和 `locales/en.json`
- **引擎 prompt**：完整中文版 + 完整英文版，按 `config.language` 选择

**实现预估**: 3-5 天  
**文件影响**: 新增 `locales/` 目录，HTML 所有硬编码字符串替换

#### 建议 13: 性能优化

**现状**：大模组/长对话可能卡顿  
**目标**：流畅体验

- **虚拟滚动**：对话消息超过 100 条时用虚拟列表（只渲染可视区域）
- **懒加载仪表盘**：脉象/构成/深度 tab 按需加载数据
- **Worldbook 索引**：大世界书（1万+ 条目）建倒排索引，不用 `includes()` 暴力匹配
- **聊天截断策略**：超过 200 轮自动归档旧对话到 `.jsonl` 文件

**实现预估**: 3-5 天  
**文件影响**: 前端渲染 + `cards.js` 索引优化

#### 建议 14: 自动更新

**现状**：用户不知道有新版本  
**目标**：启动时检查新版本，一键更新

- **版本检查**：启动时 fetch GitHub 最新 release tag
- **更新提示**：「发现新版本 v2.3.0，是否下载？」
- **静默更新**：下载新 exe → 替换旧文件 → 重启

**实现预估**: 3 天  
**文件影响**: 新增 `scripts/updater.js`

## 6. 非功能建议

### 6.1 代码质量

| 项目 | 建议 | 工作量 |
|------|------|-------|
| TypeScript 迁移 | 引擎核心加类型定义 | 5-8 天 |

| API 文档生成 | JSDoc → 自动生成 API 文档 | 2 天 |
| E2E 测试 | Playwright 测试关键用户流程 | 3-5 天 |

### 6.2 安全

| 项目 | 建议 | 工作量 |
|------|------|-------|
| API Key 加密 | 用 `crypto` 模块 AES 加密存 secrets.json | 1 天 |
| CORS 限制 | 生产环境限制 `Access-Control-Allow-Origin` | 0.5 天 |
| 输入过滤 | 用户输入长度/内容安全检查 | 1 天 |

### 6.3 运维

| 项目 | 建议 | 工作量 |
|------|------|-------|
| 日志轮转 | chat.jsonl 超过 10MB 自动分割 | 1 天 |
| 健康检查 API | 新增 `/api/health` 详细诊断 | 0.5 天 |
| 启动日志 | 启动时输出端口/配置概览到 log 文件 | 0.5 天 |
