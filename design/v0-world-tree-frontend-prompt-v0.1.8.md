# v0 Prompt: World Tree v0.1.8 Frontend Redesign Prototype

请为 World Tree 设计并生成一个新的前端原型。你有一定的自主设计布局权，但必须准确覆盖下面列出的项目定位、已有功能、显示层级和交付格式。

## 项目概述

World Tree 是一个本地优先的 AI 叙事引擎 Web 控制台，当前版本目标为 `0.1.8`。它不是普通聊天应用，而是面向创作者、互动小说作者、角色扮演用户、世界观设定维护者和叙事引擎研究者的“创作者工作台”。

它把世界书、角色卡、聊天记录、叙事状态、记忆、审核队列、模型连接和本地插件组织在一起。核心价值不是“多一个 AI 聊天框”，而是让长篇互动叙事变得可管理、可观察、可回放、可调试。

产品原则：

- 本地优先：用户数据默认保存在本机。
- 中文优先：主要 UI 使用中文，关键技术词可保留英文辅助。
- 创作优先：第一屏服务继续创作、加载世界、查看世界书和存档。
- 调试后置：叙事黑盒、脉象、健康、debug 等不应压过日常创作。
- 安全边界清晰：API Key、secrets、私密运行记录、未确认素材不要出现在世界包导出默认范围里。
- 不使用内置原创素材：不要设计任何具体官方角色、故事、场景资产；可以使用抽象占位数据。

## 设计目标

设计一个新的 World Tree 前端原型，用来替代当前过多顶部 tab 的控制台。现有控制台把“工作台、角色库、世界书、对话、审核队列、连接、世界包、插件、脉象、构成、叙事黑盒、指令、体检”全部摊成同级入口，信息层级过重。新设计应把它们重新组织为更清晰的创作者工作台。

你可以自主决定视觉风格、组件排布、密度、卡片比例、颜色、图标和交互细节，但必须满足以下信息架构。

## 一级导航

请将一级导航控制在 6 个以内，建议为：

- 工作台
- 对话
- 资料库
- 世界管理
- 观测
- 设置

桌面端建议使用左侧导航或左侧 rail。移动端建议使用底部主导航，最多显示 4 个主入口：工作台、对话、资料库、设置，其余进入“更多”或抽屉。

不要继续使用 13 个横向顶部 tab。

## 首页 / 工作台

首页是最重要的页面。它应先展示“世界书”和“存档”，并支持用户通过“加载 / 继续”按钮直接在首页开启对话框。

首页必须支持两个状态：

1. 默认状态

- 主体展示当前世界书总览。
- 主体展示当前存档 / 最近存档总览。
- 显示当前世界、模型连接状态、最近游玩时间、回合数、待审核数量。
- 提供主要操作：加载并开始对话、新建世界、导入素材。

2. 加载后 / 对话状态

- 点击加载后，不跳转到完全不同的页面，而是在首页内让对话框占据主体。
- 世界书和存档从主体区域收起，变成小按钮、抽屉入口或侧边折叠入口。
- 对话框必须成为视觉中心。
- 对话状态仍能快速打开世界书、切换存档、查看叙事摘要。

首页的世界书和存档只是“展示与快速进入”，完整编辑和完整管理不放在首页。

## 对话功能

对话是核心创作区，应包含：

- 消息列表。
- 输入框。
- 发送按钮。
- 当前世界 / 当前角色上下文。
- 消息复制。
- 消息编辑。
- 消息删除。
- 消息收藏。
- 助手回复重生成。
- 候选回复 swipe / 版本选择。
- 候选回复需要表现为可持久化的版本，不要只做临时 hover 效果。
- Slash command / 命令面板入口，但不要把“指令”作为一级页面。

消息级操作应在消息 hover、更多菜单或点击后出现，不要常驻挤占聊天流。

对话页或首页对话状态的右侧 / 底部应显示叙事上下文摘要：

- 世界书命中数量。
- 当前角色状态。
- 记忆快照摘要。
- Guardian 结果摘要。
- Direction Packet 入口。
- 技术细节默认折叠。

移动端请把右侧上下文变成底部抽屉。

## 世界管理

世界管理负责项目级操作，不应和聊天主流程混在一起。

需要展示：

- 世界 / 模块列表。
- 当前世界。
- 最近世界。
- 最近存档。
- 回合数。
- 最后游玩时间。
- 分支状态。
- 新建世界。
- 删除世界。
- 从模板创建。
- 从素材创建。
- `.worldtree` 世界包导入。
- `.worldtree` 世界包导出。
- 导入前摘要确认。
- 备份入口。

`.worldtree` 世界包默认包含：

- 世界设定。
- 角色。
- 世界书。
- 默认开场。
- 来源说明。

`.worldtree` 世界包默认排除：

- API key。
- secrets。
- 用户私密 runtime 对话记录。
- memory runtime。
- state runtime。
- 未确认素材。

危险操作如删除世界、清空数据、覆盖导入必须二次确认，并且视觉上远离主按钮。

## 交互要求

- 所有主要按钮有明确状态：默认、hover、disabled、loading。
- 表单错误要靠近字段显示。
- 移动端不能横向滚动。
- 移动端触控区域至少 44px 高。
- 危险操作必须二次确认。
- 抽屉、弹窗、菜单要有关闭路径。
- 对话区和上下文区要能在桌面和移动端自然转换。
- 不要只依赖 hover，移动端也必须可操作。

## 视觉要求

你可以自主设计视觉语言，但请满足：

- 是创作者工作台，不是营销 landing page。
- 第一屏就是可用工作界面，不要做大 hero 宣传页。
- 信息密度中等偏高，但要有清晰层级。
- 可以有一点叙事感、神秘感和“世界管理”的气质，但不要牺牲可读性。
- 不要让 UI 变成普通 ChatGPT 克隆。
- 不要整页只有紫色 / 蓝紫渐变。
- 不要使用未经授权的角色图、场景图或品牌素材。
- 不要把功能说明写成大段教程文案，界面应靠结构本身说明用途。

## 交付文件格式

请交付一个可运行的前端原型，优先使用 v0 默认的 React / Next.js / Tailwind 结构。

必须输出以下文件：

- `app/page.tsx`：主页面，包含完整可切换原型。
- `app/globals.css`：全局样式和设计 token。
- `components/world-tree-shell.tsx`：整体布局、导航、顶部栏。
- `components/workbench-view.tsx`：首页 / 工作台，包括世界书、存档、加载后对话状态。
- `components/chat-view.tsx`：对话功能原型。
- `components/library-view.tsx`：资料库功能原型。
- `components/world-management-view.tsx`：世界管理功能原型。
- `components/observability-view.tsx`：观测功能原型。
- `components/settings-view.tsx`：设置功能原型。
- `lib/mock-data.ts`：所有 mock 数据集中管理。
- `README.md`：说明如何运行、页面包含哪些功能、哪些是 mock。

如果 v0 更适合输出单文件原型，也可以先输出一个完整 `page.tsx`，但必须保证组件结构清晰，后续能拆分。

不要输出：

- 后端 API。
- 数据库。
- 真实 API Key。
- 远程插件加载代码。
- 真实世界包解析实现。
- 未授权素材。

## 自主设计权限

你可以自主决定：

- 左侧导航的具体样式。
- 首页世界书和存档的布局方式。
- 对话页上下文面板的呈现形式。
- 资料库内部 tabs 的视觉样式。
- 观测页数据图表样式。
- 设置页表单布局。
- 移动端导航和抽屉交互。
- 色彩、字体、图标、卡片密度和动效。

但不能改变：

- 一级功能分组。
- 首页默认展示世界书和存档。
- 加载后首页对话框占主体。
- 世界书和存档加载后收起为小按钮 / 抽屉入口。
- 技术调试功能后置。
- API Key 和 secrets 的安全边界。
- `.worldtree` 默认排除私密数据。
- 插件只支持本地 importer / reviewer。

## 最终验收标准

生成结果应让人一眼看出：

- World Tree 是创作者工作台，不是普通聊天网页。
- 首页能先查看世界书和存档，再直接进入对话。
- 对话是核心主体。
- 资料库清楚收纳角色、世界书和审核队列。
- 世界管理清楚处理世界包、存档、备份和危险操作。
- 观测页清楚收纳叙事黑盒、脉象、构成和健康。
- 设置页清楚收纳连接、插件、数据和高级模式。
- 桌面和移动端都有合理布局。

请先生成高质量、可运行、可交互的前端原型。功能可以使用 mock 状态，不需要真实持久化。
