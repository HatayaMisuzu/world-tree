## 4. Code Findings

### CODE-P0-001: Integration tests overwrite real ignored user configuration and may overwrite secrets

- Severity: P0
- Area: test isolation / persistence / local user data
- Files: `tests/integration/helpers/server-process.js:22-31`, `tests/integration/connection-diagnostics.test.js:45-79`, `server.js:138-143`, `server.js:330-335`, `server.js:1535-1537`, `server.js:1683-1688`
- Evidence: Test helpers set only `WORLD_TREE_DATA_DIR`. `configPath()`, `secretsPath()` and `CONNECTIONS_PATH()` remain rooted at the repository's real `userData/`. The connection diagnostic test posts a fake profile with `apiKey: "test-key"` and `setDefault: true`; production code then calls `saveLlmSecret`, `saveConfig` and `saveConnectionsRaw`. `COMMAND_OUTPUTS/userData-hash-drift.tsv` proves that `userData/config.json` and `userData/connections.json` changed between the pre-command baseline and post-command state. `secrets.json` did not change in this run, but the executed code writes it; its pre-audit content already included one secret item.
- Impact: A routine `npm run test:integration` or `npm run preflight` can corrupt a user's real provider endpoint, active profile, model and secret records. This violates the audit's read-only expectation and can destroy or replace local credentials/settings.
- Why it matters: This is real local-data damage, not test flakiness. The required audit itself reproduced the mutation.
- Suggested fix: Introduce a dedicated `WORLD_TREE_USER_DATA_DIR` (or a single fully isolated app-data root), route every config/secret/connection/plugin/debug/review path through it, and make `startWorldTreeServer()` always allocate a temporary user-data directory. Add process-level refusal to run integration writes against repository `userData/`.
- Safe execution scope: `server.js` path factories, test server helper, connection/security tests, documentation of data roots. Do not touch the user's current files until the owner chooses recovery inputs.
- Tests required: A hash-invariance test around real `userData/`; temporary user-data roundtrip tests for config, connections and secrets; full integration and preflight.
- Blocks final seal? yes

### CODE-P1-001: Request-body error contract is broken and the first failure masks a second validation gap

- Severity: P1
- Area: HTTP request boundary
- Files: `src/server/http-request.js:36-42`, `src/server/http-request.js:62-70`, `tests/integration/security.test.js:76-102`
- Evidence: For oversized `Content-Length`, `readJsonBody()` rejects and immediately calls `req.destroy()`. The route catch then cannot deliver the promised JSON 413 response, so the client gets `TypeError: fetch failed`. Both the full integration run and the isolated `node --test tests/integration/security.test.js` reproduce it. Separately, successful `JSON.parse()` returns arrays/scalars unchanged; there is no object-body validation although the same integration test expects `[]` to produce `400 INVALID_JSON_BODY`. That assertion is currently masked because the earlier oversized request aborts the test.
- Impact: Clients receive a transport failure instead of a readable API error; after that is fixed, non-object JSON can reach object-oriented handlers and may be spread or interpreted unexpectedly.
- Why it matters: `npm run test:integration` is 115/116 and `npm run preflight` fails. The documented security/error contract is false.
- Suggested fix: Drain/ignore the remaining request safely or respond 413 before closing without destroying the response socket; add a JSON-object predicate at the shared request boundary (with explicit opt-in for routes that accept arrays/scalars).
- Safe execution scope: `src/server/http-request.js`, focused unit tests, `tests/integration/security.test.js`; avoid unrelated route refactors.
- Tests required: Content-Length oversize, streamed oversize, malformed JSON, array/scalar/null JSON, empty body, keep-alive follow-up request, full integration/preflight.
- Blocks final seal? yes

## 8. Final Code-Level Decision

The existing Git tag records a prior seal action; it does not change this evidence-based decision.
